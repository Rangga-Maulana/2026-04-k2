
# [HIGH] Logic flaw in `handle_action` uses post-action balances, causing 100% loss of accrued user rewards upon full withdrawal or repayment

## Severity

**Impact:** High (Permanent loss of 100% of legitimately accrued user yield/rewards)
**Likelihood:** High (Guaranteed to happen to any user performing a full withdrawal/repayment)
**Severity:** High

## Summary

The `IncentivesContract` aims to prevent flash-loan reward manipulation by calculating accrued rewards using the minimum between a user's previous balance snapshot and their current balance (`min(old_balance, new_balance)`). However, because the core token contracts (`DebtToken` and presumably `aToken`) execute state updates *before* calling the incentives controller, the `new_balance` passed during a full withdrawal or repayment is exactly `0`. Consequently, the mathematical evaluation becomes `min(old_balance, 0) = 0`, which retroactively wipes out the user's eligible balance for the entire holding period, resulting in a 100% permanent loss of their accrued rewards.

## Vulnerability Details

The vulnerability stems from an incorrect implementation of Aave's reward accrual mechanics. The flaw exists in the interplay between the Token contracts and the Incentives contract.

### 1. Post-Action Balance Forwarding (The Token Side)

In `contracts/debt-token/src/contract.rs`, when a user repays their debt via `burn_scaled`, the contract first updates the user's storage balance, and *then* calls `handle_incentives_action`:

```rust
        // 1. Storage is updated to the new (lower/zero) balance
        storage::set_scaled_debt(&env, &on_behalf_of, new_scaled_debt); 
        // ...
        
        // 2. The incentives controller is invoked AFTER the balance is already 0
        Self::handle_incentives_action(&env, &pool_address, &borrowed_asset, &on_behalf_of, 1)?;

```

Inside `handle_incentives_action`, the code reads the new balance (`let user_balance = storage::get_scaled_debt(env, user);`) and forwards this post-action balance to the Incentives contract.

### 2. The Flawed Anti-Flashloan Logic (The Incentives Side)

In `contracts/incentives/src/contract.rs` within the `handle_action` function, the developers attempted to prevent flash-loan abuse by implementing the following logic:

```rust
        // WARDEN: The token contract passed `user_balance` = 0 (post-repayment)
        // user_data.balance_snapshot = the user's historical holding balance

        let balance_for_rewards = if user_data.balance_snapshot == 0 {
            0
        } else {
            // [@AUDIT-BUG] Flawed logic: min(old_balance, 0) always equals 0!
            user_data.balance_snapshot.min(user_balance)
        };

        let new_accrued = calculation::calculate_user_accrued_rewards(
            &env,
            updated_index.index,
            user_data.index_snapshot,
            balance_for_rewards, // Evaluates to 0
        )?;

```

Because the user is performing a full repayment (or full withdrawal of collateral), `user_balance` is `0`. The `min()` function selects `0` as the `balance_for_rewards`. The user's time-weighted yield over the past days/months/years is multiplied by `0`, burning all their pending rewards instantly.

## Proof of Concept

The following end-to-end test proves the vulnerability using the actual compiled WASM contracts (Real State execution). It simulates a user holding a borrow position for 1 year, accruing massive rewards, and then losing all of it upon repayment.

Place this inside `lib.rs`:

```rust

#[test]
fn test_withdrawal_penalty_logic_flaw_real_state() {
    let env = soroban_sdk::Env::default();
    let setup = Setup::new(&env);
    
    // Override auth for injection
    env.mock_all_auths();

    std::println!("\n===============================================");
    std::println!("[+] PHASE 1: REAL INCENTIVES DEPLOYMENT & SETUP");
    std::println!("===============================================");

    // 1. Deploy the actual Incentives WASM (No Mocks)
    let incentives_addr = env.register(incentives::WASM, ());
    let incentives_client = incentives::Client::new(&env, &incentives_addr);
    incentives_client.initialize(&setup.admin, &setup.router_addr);

    // 2. Configure Rewards for Borrowing Asset B
    let reward_token = setup.asset_a.clone(); // Using Asset A as the payout token
    let emission_rate = 1000u128; // 1000 tokens per second
    
    incentives_client.configure_asset_rewards(
        &setup.admin,
        &setup.debt_token_b, // Asset parameter = token calling handle_action
        &reward_token,
        &1u32,                // REWARD_TYPE_BORROW
        &emission_rate,
        &0u64,               // No distribution end
    );

    // 3. Inject Real Incentives into DebtToken Asset B
    env.invoke_contract::<()>(
        &setup.debt_token_b,
        &soroban_sdk::Symbol::new(&env, "set_incentives_contract"),
        soroban_sdk::vec![&env, setup.router_addr.to_val(), incentives_addr.to_val()],
    );

    std::println!("[*] Real Incentives Contract deployed and wired.");

    std::println!("\n===============================================");
    std::println!("[+] PHASE 2: NORMAL FLOW (DEPOSIT & BORROW)");
    std::println!("===============================================");
    
    let deposit: u128 = 10_000_000_000; 
    setup.router.supply(&setup.user, &setup.asset_a, &deposit, &setup.user, &0u32);

    let borrow_amount: u128 = 5_000_000_000;
    setup.router.borrow(
        &setup.user,
        &setup.asset_b,
        &borrow_amount,
        &1u32, 
        &0u32, 
        &setup.user,
    );
    std::println!("[*] User borrowed {} units of Asset B.", borrow_amount);

    std::println!("\n===============================================");
    std::println!("[+] PHASE 3: TIME TRAVEL (ACCRUING YIELD)");
    std::println!("===============================================");

    // Advance the Soroban ledger by 1 Year (31,536,000 seconds)
    let current_timestamp = env.ledger().timestamp();
    let one_year = 31_536_000;
    
    // Simulate time passing to accrue massive rewards
    env.ledger().set(soroban_sdk::testutils::LedgerInfo {
        timestamp: current_timestamp + one_year,
        sequence_number: env.ledger().sequence() + 10000,
        ..env.ledger().get()
    });

    let expected_rewards = emission_rate * (one_year as u128);
    std::println!("[*] Fast forwarded 1 year. Protocol owes user ~{} in rewards.", expected_rewards);

    std::println!("\n===============================================");
    std::println!("[+] PHASE 4: THE PENALTY TRIGGER (REPAY)");
    std::println!("===============================================");
    std::println!("[*] User executes a full repayment to claim their 1-year yield.");

    // User repays debt. This triggers `handle_action` with a new balance of 0.
    setup.router.repay(
        &setup.user,
        &setup.asset_b,
        &borrow_amount,
        &1u32,
        &setup.user,
    );

    std::println!("\n===============================================");
    std::println!("[+] PHASE 5: POST-MORTEM STATE FORENSICS");
    std::println!("===============================================");
    
    // Read the exact state from the deployed Incentives contract
    let user_reward_data = incentives_client.get_user_reward_data(
        &setup.debt_token_b,
        &reward_token,
        &setup.user,
        &1u32, // REWARD_TYPE_BORROW
    );

    std::println!("[!] Accrued Rewards in Contract State : {}", user_reward_data.accrued);
    std::println!("[!] Expected Accrued Rewards          : > {}", expected_rewards);
    
    if user_reward_data.accrued == 0 {
        std::println!("[!] IMPACT VERIFIED: 100% loss of yield due to post-action snapshotting flaw.");
    }

    // Strict assertion to validate the vulnerability
    assert_eq!(user_reward_data.accrued, 0, "Exploit Failed: Rewards were not wiped.");
}
```

## Recommended Mitigation Steps

Aave v3 handles this correctly by strictly separating the previous balance used for past accrual from the new balance used for future accrual.

1. Calculate the pending rewards for the elapsed time using `user_data.balance_snapshot` directly (do **not** use `min(snapshot, new_balance)`).
2. Add the result to `user_data.accrued`.
3. Update `user_data.balance_snapshot = new_balance` for the *next* accrual period.

```rust
        // Correct implementation
        let balance_for_rewards = user_data.balance_snapshot; // Always use the historical balance!
        
        let new_accrued = calculation::calculate_user_accrued_rewards(
            &env,
            updated_index.index,
            user_data.index_snapshot,
            balance_for_rewards,
        )?;

        user_data.accrued = user_data.accrued.checked_add(new_accrued)?;
        user_data.index_snapshot = updated_index.index;
        user_data.balance_snapshot = user_balance; // Save post-action balance for the NEXT interaction

```
