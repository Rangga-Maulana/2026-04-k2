# Missing access control on `refresh_prices` allows attackers to bypass the Oracle circuit breaker and drain protocol reserves

## Severity

**Impact:** High (Theft of protocol funds, total insolvency)
**Likelihood:** High (Can be executed deterministically in a single transaction)
**Severity:** Critical

## Summary

A critical access control flaw in the `PriceOracleContract` allows an attacker to artificially inflate the price of any collateral asset. By exploiting the public nature of the `refresh_prices` function, an attacker can bypass the oracle's 20% circuit breaker limit through "creeping price manipulation" (incrementally shifting the baseline price). This allows the attacker to borrow massive amounts of underlying assets against heavily inflated, worthless collateral, draining the protocol and leaving it with permanent bad debt.

## Vulnerability Details

The K2 protocol relies on a circuit breaker mechanism within the oracle to prevent extreme price volatility (e.g., flash loan manipulation on DEXes). It enforces a maximum price deviation (`max_price_change_bps`, nominally 20%) against a baseline `last_price`.

However, the architecture contains two fatal flaws:

1. **Missing Access Control:** The `refresh_prices` function is public. Anyone can call it.
2. **Instant Anchor Shifting:** When `refresh_prices` is called, it clears the price cache TTL and fetches the current spot price. If the new spot price is within the 20% limit of the old baseline, it accepts the new price and **instantly overwrites `last_price` with the new value**, without any time-weighted delay or block cooldown.


## Vulnerable Code Location

The vulnerability is rooted in a toxic combination of two specific code sections inside the `PriceOracleContract` contract.

### 1. Missing Access Control in `refresh_prices`
- **File:** `contracts/price-oracle/src/contract.rs`
- **Function:** `refresh_prices`

The function is declared as a public entry point but lacks any access control modifier or `require_auth()` check. This allows any unprivileged user or malicious smart contract to trigger a cache wipe and force-fetch a price update at will.

```rust
    /// Force-refresh cached prices for specific assets.
    /// Clears cached price data and fetches fresh prices from external sources.
    pub fn refresh_prices(env: Env, assets: Vec<Asset>) -> Result<Vec<PriceData>, OracleError> {
        if Self::is_paused(env.clone()) {
            return Err(OracleError::OracleQueryFailed);
        }

        // [@AUDIT-BUG-1] No access control or require_auth() here. Completely public.
        // Clear cache for requested assets so fetch is forced
        for asset in assets.iter() {
            storage::clear_last_price_data(&env, &asset);
        }

        // Fetch fresh prices (will bypass cache since we just cleared it)
        Self::get_asset_prices_vec(env, assets)
    }

```

### 2. Instant Circuit Breaker Anchor Shifting

* **File:** `contracts/price-oracle/src/contract.rs`
* **Function:** `get_asset_price_data_with_config`

When a new price is fetched via the standard path, it is passed into `validate_price_change`. If the price movement is under the configured threshold (e.g., a 9% change against a 20% max cap), the function returns `Ok(())`, and the system **instantly updates the baseline anchor** via `storage::set_last_price` without any time-weighted filter or block cooldown constraint.

```rust
        // ... (Price resolution via custom or external oracles) ...

        // Validates price deviation against the baseline last_price
        Self::validate_price_change(&env, &asset, &price_data.price, oracle_config)?;
        
        // [@AUDIT-BUG-2] Instant baseline shifting occurs here within the same transaction.
        // It permanently updates the circuit breaker anchor to the manipulated spot price.
        storage::set_last_price(&env, &asset, price_data.price);
        storage::set_last_price_data(&env, &asset, &price_data);
        
        Ok(price_data)
    }

```

### The Attack Path (Optimized for Soroban VM)

Because the Soroban VM imposes a strict 100M CPU instruction limit per transaction, an attacker cannot perform an infinite loop. However, they only need a few iterations to break the protocol:

1. The attacker takes a Flash Loan and targets a specific collateral asset's DEX pool (e.g., Soroswap).
2. Inside the Flash Loan callback, the attacker swaps tokens to push the spot price up by **9%** (safely below the 20% circuit breaker threshold).
3. The attacker calls `oracle.refresh_prices()`. The protocol accepts the 9% jump and sets the new baseline.
4. The attacker repeats steps 2 and 3 approximately **10 times** within the same ledger sequence.
5. Compounding 9% over 10 iterations inflates the asset's price by over **236%**. This perfectly fits within Soroban's 100M CPU budget.
6. The attacker supplies a small amount of the inflated asset and borrows the entire capacity of another reserve (e.g., USDC/USDT), successfully draining the pool.

## Impact

The impact of this vulnerability is Critical, leading to the complete theft of protocol reserves and immediate systemic insolvency.

By chaining multiple underlying price manipulations (e.g., +9% jumps via DEX swaps) and repeatedly invoking the unprotected `refresh_prices` function, an attacker completely circumvents the oracle's 20% circuit breaker protection. This architectural flaw allows a malicious actor to artificially inflate the oracle-perceived value of their collateral to virtually any amount within a single ledger sequence, easily fitting within the Soroban VM's 100M CPU instruction budget.

The direct financial and systemic consequences are:

1. Theft of Funds (Pool Drain): The attacker can leverage their artificially inflated Health Factor to borrow against the manipulated collateral, effectively draining 100% of the liquidity from high-value reserves (e.g., USDC, USDT) provided by legitimate depositors.

2. Systemic Insolvency: The attacker walks away with real, fully-backed assets while the protocol is left holding heavily inflated, low-value collateral.

3. Permanent Bad Debt: Because the protocol's accounting engine is entirely deceived during the transaction, the borrowed funds register as a "healthy" position. Once the spot price of the manipulated asset crashes back to its true market value, the protocol is left with massive unbacked bad debt that is mathematically impossible to liquidate profitably.

The attack is highly deterministic, requires zero initial capital (can be executed entirely via flash loans), and causes catastrophic loss of user funds.

## Recommended Mitigation Steps

To secure the oracle and the accounting engine, implement the following two layers of defense:

1. **Strict Access Control:** Restrict the `refresh_prices` function. It should only be callable by authorized backend keeper bots or via `admin.require_auth()`. Do not allow public users to force cache flushes.
2. **Time-Weighted Anchor (Cooldown):** Do not update `last_price` (the circuit breaker baseline) instantaneously. Implement a minimum time delta (e.g., `timestamp_now - last_update_timestamp > 1_hour`) before a new spot price can become the new baseline, or utilize an Exponential Moving Average (EMA) to resist rapid, same-ledger price inflation.
